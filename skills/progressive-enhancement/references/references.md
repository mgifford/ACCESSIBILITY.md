- **HTML5 Doctor** — <https://html5doctor.com>

## WCAG Success Criteria

- **WCAG 1.3.1 Info and Relationships (Level A)** — <https://www.w3.org/WAI/WCAG22/Understanding/info-and-relationships.html>  
  Semantic structure must be conveyed programmatically, not only visually.

- **WCAG 2.1.1 Keyboard (Level A)** — <https://www.w3.org/WAI/WCAG22/Understanding/keyboard.html>  
  All functionality must be operable without JavaScript-only interaction paths.

- **WCAG 4.1.1 Parsing (Level A)** — <https://www.w3.org/WAI/WCAG22/Understanding/parsing.html>  
  HTML must be well-formed so the baseline layer is reliable before enhancements apply.

- **WCAG 4.1.2 Name, Role, Value (Level A)** — <https://www.w3.org/WAI/WCAG22/Understanding/name-role-value.html>  
  Custom components built on progressive enhancement must expose accessible name, role, and state to assistive technology.

## Design Principles

- **The Rule of Least Power** — <https://www.w3.org/2001/tag/doc/leastPower.html>  
  W3C TAG finding: use the least powerful language suitable for the job; prefer HTML over CSS over JavaScript where any suffices.
